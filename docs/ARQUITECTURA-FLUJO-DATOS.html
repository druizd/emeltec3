Recorrido del dato

📥 Ingesta

  1. El MT genera CSV/log con el id_serial en el tagname.
  2. Llegan al Windows Server por la VPN de Emeltec a data/incoming_logs/.
  3. csvprocessor extrae el serial, respalda el raw, transforma a TelemetryRecord y lo deja pending en SQLite.
  4. Lo envía por gRPC a csvconsumer (:50051), que inserta en la base de datos.
  5. Si el envío falla, queda pending y se reintenta; a las 3 fallas pasa a failed_logs.

🌐 Consulta / web

  • El usuario entra por Nginx (TLS, *.emeltec.cl), que sirve el frontend y enruta /api/ → main-api y /api/auth/ → auth-api.
  • main-api lee la base de datos, usa Redis como caché de "online", valida sesión con auth-api (JWT) y consulta valores en vivo por gRPC a csvconsumer.

⚙️ Actuación (comando PLC)

  • main-api encola un comando en linux-db-api (:3010).
  • El consumidor que lo escribe en el PLC y el recorrido de vuelta al MT deben confirmarse con operaciones verificar.

🔒 Límites de seguridad

  • Borde: Nginx termina TLS y publica solo *.emeltec.cl; el resto queda en 127.0.0.1 detrás del proxy.
  • Ingesta cross-host: los consumidores y linux-db-api reciben tráfico del Windows Server → firewall/NSG + (pendiente) auth/TLS gRPC.
  • Autorización: todo acceso por id_serial/sitio pasa por el control multi-tenant (canAccessSite).
